GitHub

A reference page for how to use GitHub for version control.


What is GitHub and why use it?

GitHub is one of many platforms that provide a convenient web interface for Git, an open-source version control system. Version control with GitHub has many benefits, including:
  • The ability to easily back-up and rollback changes to your personal projects.
  • The ability to access your work across different computers or other platforms.
  • Fascilitating collaorative development and the sharing of code.
  • Allowing for the publication of websites such as this one.

GitHub can be used in many ways, including through its website interface and an integrated GUIs in editors like VS Code. Here I give some basic details and reference commands for how to work with GitHb through the command line.

For details on how to install git check here.
Atlassian also has great detailed tutorials on the Git workflow here.


GitHub command references

git config
See values for configuration settings. E.g., git config user.email
Edit values of configuration settings. E.g., git config --global user.email "your_email@example.com"
The --local flag changes the values just for this repository in the .git/config file (default behaviour).
The --global flag changes the values on an operating system level in the .gitconfig file in the user's home directory.


git init
git init <directory>git init --bare <directory.git>
Turns the currecnt directory into a Git repository, or creates a new git repository at the specified directory.
The --bare flag is typically used to set up a central repository of a shared project.

git clone <repo>
git clone <repo> <directory>git clone --branch <tag> <repo>
Creates a copy of an existing repository.
Use git clone --branch <tag> to specify the branch to copy.


git add
git add <filename>git add <directory>git add -Agit add .
git add -ugit add -p
Adds changes from the working directory to the staging area.
Stage all (new, modified, deleted) files git add -A
Stage all (new, modified, deleted) files in a directory git add <directory>
Stage modified and deleted files only (not new files) git add -u
Begin an interactive staging sessiong git add -p
Files and filetypes you don't want added can be specified in your .gitignore.

git commit
git commit -m <message>git commit --ammend
Captures a snapshot of the project's currently staged changes.
Use -m to add a commit message in the command window.
The --a flag automatically stages modified or deleted files (but not new files).
The --amend flag is used to ammend the previous commit. Use --no-edit to avoid updating the commit message.

git status
Displays the state of the working directory and the staging area
List which files are staged, unstaged, and untracked.

git diff
git diff HEADgit diff --cached
Displays differences between two data sets (can be files, commits, branches, etc).
git diff shows uncommitted changes since the last commit.
git diff HEAD shows differences between the working directory and the last commit.
git diff --cached show differences between staged changes and the last commit

git log
--graph --pretty=format:'%Cred%h%Creset -%C(yellow)%d%Creset %s %Cgreen(%cr) %C(bold blue)<%an>%Creset' --abbrev-commit
Displays the commit history, with nice formatting if you use the fancy flags above (could be made an alias).


git clean
git clean -ngit clean -fgit clean -d
Works as an undo on untracked files. Default is non-recursive through directories.
The -n flag performs a dry run and displays which files would be removed.
Requires -f flag to be applied to force deletion.
The -d flag allows removal of untracked directories..

git revert
git revert <commit ref>
Performs a new commit that undoes any previous commit
git revert HEAD is used to revert to the previous commit

git reset

--soft, --mixed, --hard
Moves both the HEAD and branch refs to a specified commit.
N.B. Reverting can cause issues in shared repositories


git remote
git remote -vgit remote add <name> <url>
Lets you create, view, and delete connections to other repositories, serving as a placeholder in .git/config for the full URL.
git remote lists the remote connections to other repositories.
git remote -v includes the URL of each connection.
git remote add <name> <url> creates a new connection to a remote repository.
git remote rm <name> removes the connection to a remote repository.
git remote rename <old_name> <new_name> renames a remote connection


git fetch
git fetch <remote>git fetch <remote> <branch>
Primary command to download commits, files, and refs from a remote repository into a local repo.
Safer than git pull as it downloads the remote content but does not update the local repo's working state.
git fetch <remote> downloads all branches, commits and files from a repository.
git fetch <remote> <branch> fetches only the specified branch.
The --dry-run flag outputs examples of actions that would be performed but not apply them.

git pull
git pull --rebasegit pull <remote>git pull --verbose
The git pull command is actually a combination of two other commands, git fetch followed by git merge.
Use --rebase flag to implement a rebase merging strategy instead of a merge commit.

git push
git push <remote> <branch>
Used to transfer commits from your local repository to a remote repository.

git branch
git branch <branch>git branch -a
The git branch command lets you create, list, rename, and delete branches.
Each branch has its own working directory, staging area, and project history.
A new branch shouuld be used wheneveer adding a new feature or fixing a bug.
New commits are recorded in the history for the current branch, which results in a fork in the history of the project.
git branch lists all of the local branches in a repository. Equivalent to git branch --list.
git branch -a lists all of the remote branches for a repository.
git branch <branch> creates a new branch. Run git checkout <branch> after to switch to this branch.

git checkout
git checkout <branch>git checkout -b <newbranch>git checkout -b <newbranch> <existingbranch>
Updates the files in the working directory to match the version stored in that branch, and tells Git to record all new commits on that branch.
git checkout <branch> switches to a branch, pointing HEAD to the tip of that branch.
git checkout -b <newbranch> creates a new branch and switches to that branch.
git checkout -b <newbranch> <existingbranch> bases the new branch off a different base, instead of HEAD.

git merge
git merge <branch>git merge <branch> --no-ff
Combines another branch into the current branch.
Before merging:
  • Execute git status to ensure that HEAD is pointing to the correct merge-receiving branch.
  • Execute git checkout to switch to the receiving branch if not on it already.
  • Execute git fetch to pull the latest remote commits.
  • Once the fetch is completed ensure the main branch has the latest updates by executing git pull.
git merge <branch> merges that branch into the receiving branch.
Pass the --no-ff flag to create a new commit that documents this merge.
If a merge conflict occurs:
  • Run git status to show which files need to be resolved.
  • Resolve differences between files.
  • Run git add on the conflicted file(s) to tell Git they're resolved.
  • Run a normal git commit to generate the merge commit.


git rebase
git rebase <base>git rebase -i <base>
Moves or combines a sequence of commits to a new base commit.
Never rebase commits once they've been pushed to a public repository.
git rebase <base> automatically rebases the current branch onto <base>
Pass the -i flag to rebase interactively.
